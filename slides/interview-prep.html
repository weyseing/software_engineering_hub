Chapter 4 · Hard Problems

Exactly-Once Payment Processing

At [COMPANY/PRODUCT] — the hardest constraint in distributed payment systems: guarantee money moves exactly once, even when networks lie.

The Problem

Retries That Double-Charge

Payment provider timeouts caused client retries. Without idempotency, a timed-out request that actually succeeded would charge the customer twice. Every retry was a liability.

Why It Was Hard

Distributed State, No Coordination

The payment provider, our service, and the client all had independent views of state. A "failed" response could mean: never received, received and failed, or received and succeeded but response lost.

Stakes: A single double-charge incident generates support tickets, refund costs, and merchant trust damage — at scale, this was a multi-thousand dollar risk per outage window.
Chapter 4 · Hard Problems

The Idempotency Solution

A client-generated key that survives retries, provider failures, and network partitions — making every payment operation safe to replay.

Step 1 — Key

Client-Generated Idempotency Key

UUID v4 generated at the client for each payment intent. Stored with TTL in Redis. Any retry with the same key returns the cached result — the provider is never hit twice.

›
Step 2 — State

Explicit Payment State Machine

PENDING → PROCESSING → SUCCEEDED / FAILED. State transitions are atomic via DB transaction. Any request arriving in PROCESSING state waits — never duplicates.

›
Step 3 — Verify

Async Reconciliation as Safety Net

Nightly reconciliation job compares our state against provider records. Catches any gap — surfaces discrepancies before finance close, not after.

Result: Zero double-charge incidents post-rollout · Retry rate handled safely at [X]% of traffic · Reconciliation gap reduced to <0.01%
Chapter 4 · Hard Problems

RBAC at Scale — [X] Merchants, [X] Role Variants

At [COMPANY/PRODUCT] — every merchant wanted custom permissions. A flat role model collapsed. We needed hierarchy without complexity explosion.

The Problem

Role Proliferation

Started with 5 roles. Within 6 months: 40+ custom roles, inconsistent permissions, no audit trail, and engineers manually patching access in production. Unmaintainable.

Why Standard Solutions Failed

Multi-Tenant Context Breaks Simple RBAC

A "Manager" role at Merchant A should not grant any access at Merchant B. Standard role models don't scope to tenant context — we needed permissions that carried tenancy.

Risk: A misconfigured role could expose one merchant's data to another — a GDPR violation and a trust-destroying incident waiting to happen.
Chapter 4 · Hard Problems

Role Hierarchy That Actually Scaled

A three-layer model: global roles define capability, tenant roles scope it, resource permissions enforce it — evaluated at every API call.

Layer 1 — Global
Capability Roles

Platform-Wide Role Definitions

Platform-level roles define what a user type can do: ADMIN, MERCHANT_OWNER, MERCHANT_STAFF, AGENT. No tenant context — pure capability definition.

Layer 2 — Tenant
Scoped Assignment

Tenant-Scoped Role Assignment

Role assignment always carries a tenant_id. A user can be MERCHANT_OWNER at tenant A and MERCHANT_STAFF at tenant B. Cross-tenant access is architecturally impossible.

Layer 3 — Resource
Permission Enforcement

API-Layer Permission Check

Every endpoint declares required permissions. Middleware resolves: user → tenant-scoped role → permissions → allow/deny. Fully audited, cached in Redis per session.

Result: Role count from 40+ → 5 canonical roles · Zero cross-tenant access incidents · New merchant onboarded in <2 minutes with correct permissions auto-assigned

Chapter 5 · Engineering Foundation

Test Strategy for Payment Systems

Three layers with different purposes — not coverage theatre, but confidence at each integration boundary.

Unit Layer 1

Business Logic Isolation

Payment calculations, RBAC evaluation, idempotency key logic. Fast, deterministic, runs on every commit. No I/O — pure domain logic. Target: >80% coverage on core modules.

Contract Layer 2

API Contracts (Pact)

Consumer-driven contracts between our payment service and provider adapters. Catches breaking changes before staging deploy. Runs on every MR — no integration environment needed.

Load Layer 3

Performance Under Pressure

k6 load tests simulate merchant traffic spikes. Run weekly and before major releases. Baseline: payment flow at 10× normal load. Any regression in p99 blocks the release.

PCI context: Test data fully synthetic — no real card numbers in any environment below production. Enforced at the pipeline level, not by convention.

Chapter 6 · Critical Incidents

Blameless Post-Mortem

The incident is only half the work. The post-mortem converts a production failure into a structural, permanent improvement to the system.

Step 01

What Happened

Timestamped event reconstruction. No blame. Focus exclusively on system and process failures — not individual mistakes or judgement calls made under pressure.

Step 02

Root Cause Analysis

5 Whys or fishbone analysis. Distinguish the proximate trigger from the underlying systemic cause that allowed the failure to reach production.

Step 03

Action Items

Assigned owners, firm deadlines, tracked as GitLab issues. Progress reviewed in the following sprint retro without exception.

Principle: Every incident is a signal — it reveals exactly where the system and process need to improve. Treat it as an investment, not a cost.
